# 外籍旅客配送平台－系統分析（MVP 初稿）

文件版本：0.2  
建立日期：2026-07-13  
文件狀態：需求分析初稿／待業務確認

## 1. 文件目的

本文件依據目前已知需求，分析外籍旅客配送平台的業務目標、使用者角色、核心流程、功能模組、資料需求、外部介接、例外情境及 MVP 開發範圍，作為需求訪談、畫面設計、API 設計及程式開發的共同基礎。

目前原始需求資訊有限，本文中未經確認的內容均標示為「建議」或「待確認」，不可直接視為最終合約需求。

## 2. 原始需求

希望建置一個外籍旅客配送平台。配送貨物來源為「E指購後台」中的購物訂單；符合機場配送條件的訂單由 E指購後台傳送至配送平台，再透過中華郵政由商品出貨店家寄送至機場，並透過通關金流完成配送相關費用付款。

本版暫定 E指購後台為「訂單與商品資料的主來源」，配送平台不重複建立購物訂單，只建立與管理配送任務。最終資料主從關係仍需由雙方系統負責人確認。

## 3. 系統目標

1. 接收 E指購後台提供的購物訂單、旅客、商品與出貨店家資料。
2. 將 E指購訂單轉換為可追蹤的機場配送任務，避免重複建單。
3. 串接金流，完成運費或相關服務費付款。
4. 串接或記錄中華郵政託運資訊，追蹤配送狀態。
5. 讓店家人員確認出貨資料並完成交寄。
6. 讓機場人員可辦理到貨、保管及旅客取貨。
7. 讓旅客可查詢付款、配送與取貨狀態。
8. 將必要的配送結果回傳 E指購後台。
9. 讓管理者可查詢訂單、處理異常並保留操作紀錄。

## 4. 系統範圍

### 4.1 MVP 納入範圍

- 接收 E指購後台傳入的配送需求，並以來源訂單編號避免重複建立。
- 店家查詢、確認及補充配送資料；原始購物訂單由 E指購後台維護。
- 接收旅客、商品、出貨店家等來源資料，並補充航班與取貨資料。
- 商品件數與包裹基本資料確認或補登。
- 配送費用試算或人工輸入。
- 金流付款及付款結果記錄。
- 中華郵政託運單號記錄。
- 配送狀態查詢與人工更新；若已有可用 API，再納入自動同步。
- 機場到貨、保管、取貨核銷。
- QR Code 或取貨碼驗證。
- 基本後台查詢與異常註記。
- 關鍵操作稽核紀錄。
- 將配送單號、付款、物流、到貨、取貨及異常狀態回傳 E指購後台。

### 4.2 MVP 暫不納入範圍

- 多家物流商動態比價及派單。
- 自動路線最佳化。
- 完整會員、點數或行銷系統。
- 跨國配送。
- 複雜分潤與自動對帳。
- AI 商品辨識、智慧客服等延伸功能。

## 5. 使用者角色

| 角色 | 主要職責與功能 |
| --- | --- |
| 外籍旅客 | 提供旅客及航班資料、付款、查詢配送狀態、出示憑證取貨 |
| E指購後台 | 建立購物訂單，提供配送來源資料，接收配送平台處理結果 |
| 店家人員 | 查詢來源訂單、確認商品及包裹、補充出貨資料、交寄並列印相關單據 |
| 中華郵政／物流人員 | 收件、運送、提供託運單號及物流狀態；實際功能依介接方式決定 |
| 機場取貨人員 | 到貨點收、保管、核驗旅客身分、完成取貨或異常處理 |
| 平台管理者 | 管理店家、機場據點、費率、訂單、權限及異常案件 |
| 財務／客服人員 | 查詢付款、退款、配送及客訴處理紀錄 |

## 6. 核心業務流程

### 6.1 正常流程

1. 旅客在 E指購完成購物訂單，或由 E指購後台建立訂單。
2. E指購後台確認該訂單需使用機場配送服務。
3. E指購後台呼叫配送平台 API，傳送來源訂單、旅客、商品、出貨店家及配送需求。
4. 配送平台驗證資料與來源訂單編號；若已接收過相同事件則回傳既有結果，不重複建單。
5. 配送平台建立配送任務並回傳配送單號給 E指購後台。
6. 店家確認商品可配送，補充包裹、航班或取貨資料。
7. 系統檢查配送時效、必填欄位、禁運條件及服務範圍。
8. 系統計算配送費與服務費，產生待付款交易。
9. 旅客透過通關金流完成付款。
10. 系統收到付款成功通知，將訂單轉為待交寄，並回傳狀態至 E指購後台。
11. 店家產生或登錄中華郵政託運單號，完成交寄。
12. 系統取得或人工更新物流狀態，並回傳必要狀態至 E指購後台。
13. 包裹抵達機場取貨據點，機場人員完成到貨點收。
14. 系統通知旅客，並提供 QR Code 或取貨碼。
15. 旅客到機場出示護照及取貨憑證。
16. 機場人員驗證後交付包裹，系統完成取貨核銷並回傳完成狀態。

### 6.2 建議訂單狀態

| 狀態 | 說明 |
| --- | --- |
| 草稿 | 配送資料尚未完成 |
| 待資料確認 | 已由 E指購匯入，但必要配送資料尚待店家確認或補充 |
| 待付款 | 已建立費用，等待旅客付款 |
| 已付款／待交寄 | 付款成功，等待店家交寄 |
| 配送中 | 已取得託運單號並交付物流商 |
| 已抵達機場 | 機場據點已完成到貨點收 |
| 待取貨 | 包裹可供旅客領取 |
| 已取貨 | 身分驗證及交付完成 |
| 已取消 | 尚未完成配送即取消 |
| 配送異常 | 遺失、破損、延誤、錯誤配送等 |
| 逾期未取 | 超過保管期限仍未領取 |
| 退運中／已退運 | 包裹進入或完成退運 |

狀態應以「狀態轉換規則」控制，不允許使用者任意跳轉，並保存每次變更的時間與操作者。

## 7. 功能需求分析

### 7.1 帳號與權限

- 使用者登入、登出及密碼重設。
- 角色權限控制：店家、機場、客服、財務、管理者。
- 店家人員只能存取所屬店家的訂單。
- 機場人員只能存取所屬取貨據點的資料。
- 敏感操作需保存操作者、時間、操作內容及來源。

### 7.2 店家與據點管理

- 店家基本資料、統一編號、聯絡資訊及啟用狀態。
- 店家人員帳號與所屬關係。
- 支援機場、航廈、取貨櫃檯及營業時間設定。
- 設定各據點可接受的配送條件與最晚到貨時間。

### 7.3 E指購訂單介接

- 提供 E指購建立配送需求的 API。
- 接收來源系統代碼、來源訂單編號、訂單時間、店家、旅客、商品、金額及配送需求。
- 使用「來源系統代碼＋來源訂單編號」建立唯一性限制，避免重複建立配送任務。
- 每次請求須包含事件識別碼或冪等鍵，使 E指購逾時重送時可安全取得相同結果。
- 驗證必填欄位、格式、店家對照、商品配送資格與資料版本。
- 接收來源訂單取消、商品異動或旅客資料異動事件，依配送階段決定自動更新或轉人工處理。
- 回傳配送單號、接收結果與錯誤明細。
- 提供狀態回傳 API 或事件通知，將付款、交寄、到貨、取貨、取消及異常結果通知 E指購。
- 保存每次傳入、回傳、重試及失敗結果；日誌中的個資與金鑰必須遮罩。

### 7.4 配送單管理

- 由 E指購來源訂單建立、暫存、補充、取消及查詢配送單。
- 系統產生唯一配送單號。
- 保存來源系統、來源訂單編號、來源訂單版本與最近同步時間。
- 原始商品、旅客或訂單資料原則上由 E指購維護；配送平台只維護配送專屬資料。
- 登錄旅客姓名、國籍、護照識別資訊、聯絡方式與偏好語言。
- 登錄航班日期、航班編號、起飛時間、機場、航廈及取貨據點。
- 登錄商品名稱、類別、數量、金額及禁運檢核結果。
- 登錄包裹件數、重量及尺寸。
- 記錄店家備註、旅客備註及內部處理備註。
- 查詢條件至少包含配送單號、託運單號、旅客、店家、航班日期及狀態。

### 7.5 費用與付款

- 依重量、尺寸、配送區域或費率表計算費用；實際規則待確認。
- 顯示運費、服務費、稅額、優惠及應付總額。
- 產生金流付款交易，避免重複付款。
- 接收金流回呼並驗證交易結果。
- 保存金流交易編號、付款方式、金額、幣別、付款時間及狀態。
- 支援付款失敗後重新付款。
- 退款必須有原因、審核及完整紀錄；MVP 可先採後台人工發起。

### 7.6 郵政配送

- 建立或登錄託運資訊與託運單號。
- 若中華郵政提供可用 API，支援建立託運單、列印標籤及查詢物流軌跡。
- 若無 API，MVP 採人工登錄託運單號及狀態。
- 託運單號不可重複綁定不同有效訂單。
- 保存交寄時間、預計到貨時間及實際到貨時間。

### 7.7 機場到貨與取貨

- 依託運單號或配送單號進行到貨點收。
- 記錄包裹外觀、件數、存放位置及異常情形。
- 到貨後產生或啟用一次性取貨碼／QR Code。
- 取貨時核對護照、航班與取貨憑證。
- 取貨碼使用後立即失效，避免重複領取。
- 保存交付人員、交付時間及驗證結果。

### 7.8 通知

- 付款成功通知。
- 已交寄及物流狀態通知。
- 已抵達機場／可取貨通知。
- 航班前尚未到貨或即將逾期通知。
- 異常、取消、退款及退運通知。
- MVP 建議先支援 Email；簡訊、LINE 或其他通訊服務視需求擴充。

### 7.9 後台與報表

- 配送單、付款、退款及物流狀態查詢。
- 異常案件建立、指派、備註及結案。
- 每日新增、付款成功、交寄、到貨及取貨件數統計。
- 支援 CSV／Excel 匯出；敏感資料應遮罩並受權限控制。

## 9. 外部系統介接

### 9.1 E指購後台

需取得並確認：

- E指購測試與正式環境、認證方式、來源 IP 或憑證規則。
- 來源訂單、店家、旅客、商品、付款及配送資料欄位。
- 哪一套系統負責蒐集航班、護照識別資料與機場取貨資訊。
- E指購訂單成立、異動、取消的事件時機與資料版本規則。
- 配送平台需回傳哪些狀態，以及採即時回呼或由 E指購主動查詢。
- API 逾時、重送、亂序、重複事件及兩邊狀態不一致時的處理方式。
- 來源訂單編號是否跨店家唯一；若否，唯一鍵須加入店家代碼。
- E指購訂單與配送單為一對一或一對多，例如一筆訂單由多個店家分別出貨。

### 9.2 通關金流

需取得並確認：

- 測試與正式環境網址。
- 商店代號、API 金鑰與簽章規則。
- 建立付款、查詢交易、取消及退款 API。
- 付款完成回呼格式與重送機制。
- 支援付款方式、幣別、手續費及結算方式。
- 超時、重複通知及金額不一致的處理規則。

### 9.3 中華郵政

需取得並確認：

- 是否提供企業客戶建單、標籤列印與物流查詢 API。
- 認證方式、測試環境、呼叫限制與錯誤碼。
- 託運標籤格式及印表機需求。
- 重量、尺寸、禁運品、理賠及退運規則。
- 物流狀態代碼與平台狀態的對照關係。

### 9.4 航班資訊（建議第二階段）

MVP 可由店家人工輸入航班資料；若需自動驗證航班時間，另行串接航班資料服務，並評估授權費用及資料正確性。

## 10. 業務規則

1. 配送平台只接受已授權的 E指購系統及已啟用店家送入資料。
2. 相同來源訂單不得產生重複有效配送單；若支援分批出貨，須使用來源訂單明細或出貨批次識別。
3. E指購負責購物訂單資料，配送平台負責配送任務資料；跨系統更新須依資料主責規則處理。
4. 配送服務只接受已啟用店家與取貨據點。
5. 建立配送任務前須檢查航班時間是否滿足配送所需安全時間。
6. 未付款訂單不得進入正式交寄流程；是否允許店家代墊待確認。
7. 實際付款金額必須與平台訂單金額一致。
8. 同一筆 E指購事件、金流回呼及物流事件必須可重複接收，但只能處理一次。
9. 已交寄訂單不可因 E指購取消事件直接刪除，應轉人工處理、退款或退運。
10. 取貨需同時驗證取貨憑證及旅客身分；代理領取規則待確認。
11. 超過保管期限未取貨時轉為逾期未取，後續依政策退運或處置。
12. 所有費用、狀態、退款、來源異動與人工更正均需保留歷程，不得直接覆蓋而無紀錄。

## 11. 例外與風險情境

| 情境 | 建議處理方式 |
| --- | --- |
| 付款失敗或逾時 | 保留待付款狀態，允許重新產生付款交易 |
| 金流重複回呼 | 以交易編號與事件識別碼進行冪等處理 |
| 付款成功但平台未更新 | 由排程主動查詢交易並補償更新 |
| E指購重複送單 | 以來源訂單唯一鍵與冪等鍵回傳既有配送單，不重複建立 |
| E指購送單逾時 | E指購可使用相同冪等鍵重送，或用來源訂單編號查詢結果 |
| E指購訂單異動 | 未交寄時依主責規則更新；已交寄後轉人工異常處理 |
| E指購與配送平台狀態不一致 | 由排程對帳並產生異常清單，不直接覆蓋終態資料 |
| 物流 API 暫時失敗 | 重試並記錄失敗；提供人工補登功能 |
| 航班提前或異動 | 由客服調整取貨資料並留下異動紀錄 |
| 無法在起飛前送達 | 通知旅客與客服，依政策取消、退款或改送地點 |
| 包裹破損、遺失或件數不符 | 建立異常案件，保留照片、責任與處理結果 |
| 旅客未取貨 | 到期提醒，之後轉入退運或保管費流程 |
| 取貨碼外洩 | 仍須核對護照；必要時作廢並重發取貨碼 |

## 12. 非功能需求

### 12.1 安全與個資

- 全程使用 HTTPS。
- 角色與資料範圍權限須由伺服器端驗證。
- 密碼使用安全雜湊，不得明碼保存。
- API 金鑰、金流密鑰不得寫入原始碼。
- 護照與聯絡資料應加密、遮罩並設定保存期限。
- 查詢與匯出敏感資料需留下稽核紀錄。
- 正式上線前需確認台灣個資法、金流及跨境旅客資料相關要求。

### 12.2 可用性與效能

- MVP 建議一般查詢在正常負載下 3 秒內回應。
- 金流與物流介接失敗不得造成訂單資料遺失。
- 關鍵背景作業需具備重試、告警及人工補償機制。
- 定期備份資料庫，並演練還原流程。

### 12.3 多語系

- 旅客介面至少支援繁體中文與英文。
- 介面文字、通知範本及錯誤訊息不可硬寫於程式碼，需可維護與翻譯。
- 日期、時間、幣別與姓名格式需考量外籍旅客使用情境。

## 13. 建議系統架構

MVP 建議採模組化單體架構，降低初期部署及維運複雜度，同時保留後續拆分能力。

- 前端：店家／後台 Web；旅客可先採響應式 Web，不必立即開發 App。
- 後端：ASP.NET Core Web API（C#）。
- 資料庫：SQL Server。
- 背景工作：處理金流補查、物流同步、通知及失敗重試。
- 外部介接層：金流與中華郵政各自封裝 Adapter，避免外部格式滲入核心業務。
- E指購介接層：以獨立 Adapter 與事件收件匣保存來源事件，核心系統不直接依賴 E指購欄位命名。
- 記錄與監控：集中式應用程式日誌、介接追蹤碼與異常告警。

建議依領域區分 `SourceOrders`、`DeliveryOrders`、`Payments`、`Shipments`、`Pickups`、`Notifications`、`Identity` 模組；共用資料應透過明確介面存取，避免所有功能直接互相修改資料表。E指購 API DTO 應與核心資料模型分離，以便外部欄位調整時只修改轉換層。

## 14. MVP 驗收標準

1. 已授權 E指購後台可送入一筆資料完整的來源訂單並取得配送單號。
2. 相同來源訂單或相同冪等鍵重送時，不會產生重複配送單。
3. 無效欄位或未建檔店家會收到明確錯誤，不會建立不完整正式配送單。
4. 店家人員可查詢 E指購來源訂單並補充配送專屬資料。
5. 系統能依確認後規則產生正確應付金額。
6. 測試金流付款成功後，訂單只更新一次且留有交易紀錄。
7. 付款失敗可重新付款，不會產生重複有效付款。
8. 店家可登錄唯一託運單號並將訂單轉為配送中。
9. 機場人員可完成到貨點收及存放位置登錄。
10. 旅客可取得取貨憑證並查詢訂單狀態。
11. 機場人員核對身分與一次性取貨碼後可完成取貨；同一取貨碼不可再次使用。
12. 配送平台可依約定將付款、配送、取貨與異常狀態回傳 E指購，失敗時可重試。
13. 管理者可查詢來源訂單、配送單、付款、物流、介接事件與狀態歷程。
14. 未授權使用者無法查閱或操作其他店家／據點資料。

## 16. 建議開發順序

1. 確認 E指購資料主責、API 欄位、訂單拆分方式及狀態流程。

## 18. 結論

此平台的 MVP 核心不是重建 E指購的購物功能，而是將 E指購來源訂單可靠地轉換成配送任務，完成「來源送單、資料確認、付款、交寄、運送、機場點收、身分核驗、取貨、結果回傳」的完整閉環。現階段最重要工作是取得 E指購、金流及中華郵政介接資料，並確認跨系統資料主責、訂單拆分、航班時效、禁運品、取貨與異常處理規則；上述內容確認後，才適合進入正式畫面、API 與資料庫設計。
