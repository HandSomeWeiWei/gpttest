# 外籍旅客配送平台－系統分析（MVP 初稿）

文件版本：0.1  
建立日期：2026-07-13  
文件狀態：需求分析初稿／待業務確認

## 1. 文件目的

本文件依據目前已知需求，分析外籍旅客配送平台的業務目標、使用者角色、核心流程、功能模組、資料需求、外部介接、例外情境及 MVP 開發範圍，作為需求訪談、畫面設計、API 設計及程式開發的共同基礎。

目前原始需求資訊有限，本文中未經確認的內容均標示為「建議」或「待確認」，不可直接視為最終合約需求。

## 2. 原始需求

希望建置一個外籍旅客配送平台，將外籍旅客於各店家購買的商品，透過中華郵政寄送至機場，並透過通關金流完成付款。

## 3. 系統目標

1. 讓合作店家可快速建立外籍旅客的機場配送單。
2. 串接金流，完成運費或相關服務費付款。
3. 串接或記錄中華郵政託運資訊，追蹤配送狀態。
4. 讓機場人員可辦理到貨、保管及旅客取貨。
5. 讓旅客可查詢付款、配送與取貨狀態。
6. 讓管理者可查詢訂單、處理異常並保留操作紀錄。

## 4. 系統範圍

### 4.1 MVP 納入範圍

- 店家建立及查詢配送單。
- 旅客基本資料、航班及取貨資料登錄。
- 商品件數與包裹基本資料登錄。
- 配送費用試算或人工輸入。
- 金流付款及付款結果記錄。
- 中華郵政託運單號記錄。
- 配送狀態查詢與人工更新；若已有可用 API，再納入自動同步。
- 機場到貨、保管、取貨核銷。
- QR Code 或取貨碼驗證。
- 基本後台查詢與異常註記。
- 關鍵操作稽核紀錄。

### 4.2 MVP 暫不納入範圍

- 多家物流商動態比價及派單。
- 自動路線最佳化。
- 完整會員、點數或行銷系統。
- 跨國配送。
- 複雜分潤與自動對帳。
- AI 商品辨識、智慧客服等延伸功能。

## 5. 使用者角色

| 角色 | 主要職責與功能 |
| --- | --- |
| 外籍旅客 | 提供旅客及航班資料、付款、查詢配送狀態、出示憑證取貨 |
| 店家人員 | 建立配送單、登錄商品及包裹、收件或交寄、列印相關單據 |
| 中華郵政／物流人員 | 收件、運送、提供託運單號及物流狀態；實際功能依介接方式決定 |
| 機場取貨人員 | 到貨點收、保管、核驗旅客身分、完成取貨或異常處理 |
| 平台管理者 | 管理店家、機場據點、費率、訂單、權限及異常案件 |
| 財務／客服人員 | 查詢付款、退款、配送及客訴處理紀錄 |

## 6. 核心業務流程

### 6.1 正常流程

1. 店家確認商品可配送，並詢問旅客航班與機場取貨資訊。
2. 店家建立配送單，登錄旅客、航班、商品、包裹及取貨資料。
3. 系統檢查配送時效、必填欄位、禁運條件及服務範圍。
4. 系統計算配送費與服務費，產生待付款訂單。
5. 旅客透過通關金流完成付款。
6. 系統收到付款成功通知，將訂單轉為待交寄。
7. 店家產生或登錄中華郵政託運單號，完成交寄。
8. 系統取得或人工更新物流狀態。
9. 包裹抵達機場取貨據點，機場人員完成到貨點收。
10. 系統通知旅客，並提供 QR Code 或取貨碼。
11. 旅客到機場出示護照及取貨憑證。
12. 機場人員驗證後交付包裹，系統完成取貨核銷。

### 6.2 建議訂單狀態

| 狀態 | 說明 |
| --- | --- |
| 草稿 | 配送資料尚未完成 |
| 待付款 | 已建立費用，等待旅客付款 |
| 已付款／待交寄 | 付款成功，等待店家交寄 |
| 配送中 | 已取得託運單號並交付物流商 |
| 已抵達機場 | 機場據點已完成到貨點收 |
| 待取貨 | 包裹可供旅客領取 |
| 已取貨 | 身分驗證及交付完成 |
| 已取消 | 尚未完成配送即取消 |
| 配送異常 | 遺失、破損、延誤、錯誤配送等 |
| 逾期未取 | 超過保管期限仍未領取 |
| 退運中／已退運 | 包裹進入或完成退運 |

狀態應以「狀態轉換規則」控制，不允許使用者任意跳轉，並保存每次變更的時間與操作者。

## 7. 功能需求分析

### 7.1 帳號與權限

- 使用者登入、登出及密碼重設。
- 角色權限控制：店家、機場、客服、財務、管理者。
- 店家人員只能存取所屬店家的訂單。
- 機場人員只能存取所屬取貨據點的資料。
- 敏感操作需保存操作者、時間、操作內容及來源。

### 7.2 店家與據點管理

- 店家基本資料、統一編號、聯絡資訊及啟用狀態。
- 店家人員帳號與所屬關係。
- 支援機場、航廈、取貨櫃檯及營業時間設定。
- 設定各據點可接受的配送條件與最晚到貨時間。

### 7.3 配送單管理

- 新增、暫存、修改、取消及查詢配送單。
- 系統產生唯一配送單號。
- 登錄旅客姓名、國籍、護照識別資訊、聯絡方式與偏好語言。
- 登錄航班日期、航班編號、起飛時間、機場、航廈及取貨據點。
- 登錄商品名稱、類別、數量、金額及禁運檢核結果。
- 登錄包裹件數、重量及尺寸。
- 記錄店家備註、旅客備註及內部處理備註。
- 查詢條件至少包含配送單號、託運單號、旅客、店家、航班日期及狀態。

### 7.4 費用與付款

- 依重量、尺寸、配送區域或費率表計算費用；實際規則待確認。
- 顯示運費、服務費、稅額、優惠及應付總額。
- 產生金流付款交易，避免重複付款。
- 接收金流回呼並驗證交易結果。
- 保存金流交易編號、付款方式、金額、幣別、付款時間及狀態。
- 支援付款失敗後重新付款。
- 退款必須有原因、審核及完整紀錄；MVP 可先採後台人工發起。

### 7.5 郵政配送

- 建立或登錄託運資訊與託運單號。
- 若中華郵政提供可用 API，支援建立託運單、列印標籤及查詢物流軌跡。
- 若無 API，MVP 採人工登錄託運單號及狀態。
- 託運單號不可重複綁定不同有效訂單。
- 保存交寄時間、預計到貨時間及實際到貨時間。

### 7.6 機場到貨與取貨

- 依託運單號或配送單號進行到貨點收。
- 記錄包裹外觀、件數、存放位置及異常情形。
- 到貨後產生或啟用一次性取貨碼／QR Code。
- 取貨時核對護照、航班與取貨憑證。
- 取貨碼使用後立即失效，避免重複領取。
- 保存交付人員、交付時間及驗證結果。

### 7.7 通知

- 付款成功通知。
- 已交寄及物流狀態通知。
- 已抵達機場／可取貨通知。
- 航班前尚未到貨或即將逾期通知。
- 異常、取消、退款及退運通知。
- MVP 建議先支援 Email；簡訊、LINE 或其他通訊服務視需求擴充。

### 7.8 後台與報表

- 配送單、付款、退款及物流狀態查詢。
- 異常案件建立、指派、備註及結案。
- 每日新增、付款成功、交寄、到貨及取貨件數統計。
- 支援 CSV／Excel 匯出；敏感資料應遮罩並受權限控制。

## 8. 主要資料需求

| 資料實體 | 主要內容 |
| --- | --- |
| User／Role | 帳號、角色、所屬店家或據點、狀態 |
| Store | 店家資料、聯絡方式、服務狀態 |
| AirportPickupLocation | 機場、航廈、櫃檯、營業時間 |
| Traveler | 姓名、國籍、護照識別資料、聯絡方式、語言 |
| Flight | 航班編號、日期、起飛時間、機場、航廈 |
| DeliveryOrder | 配送單號、店家、旅客、航班、據點、費用、狀態 |
| DeliveryItem | 商品類別、名稱、數量、金額、禁運檢核 |
| Package | 件數、重量、尺寸、存放位置、外觀狀態 |
| PaymentTransaction | 金流交易編號、金額、幣別、付款狀態與時間 |
| Shipment | 物流商、託運單號、交寄及到貨時間、配送狀態 |
| PickupRecord | 取貨碼、驗證結果、領取人及交付時間 |
| StatusHistory | 對象、原狀態、新狀態、操作者、時間與原因 |
| Notification | 通知管道、收件對象、內容範本、發送結果 |
| AuditLog | 使用者、操作、對象、時間、來源與結果 |

護照等敏感資料應採最小化蒐集；若不必保存完整護照號碼，建議只保存雜湊驗證值與遮罩後末碼。

## 9. 外部系統介接

### 9.1 通關金流

需取得並確認：

- 測試與正式環境網址。
- 商店代號、API 金鑰與簽章規則。
- 建立付款、查詢交易、取消及退款 API。
- 付款完成回呼格式與重送機制。
- 支援付款方式、幣別、手續費及結算方式。
- 超時、重複通知及金額不一致的處理規則。

### 9.2 中華郵政

需取得並確認：

- 是否提供企業客戶建單、標籤列印與物流查詢 API。
- 認證方式、測試環境、呼叫限制與錯誤碼。
- 託運標籤格式及印表機需求。
- 重量、尺寸、禁運品、理賠及退運規則。
- 物流狀態代碼與平台狀態的對照關係。

### 9.3 航班資訊（建議第二階段）

MVP 可由店家人工輸入航班資料；若需自動驗證航班時間，另行串接航班資料服務，並評估授權費用及資料正確性。

## 10. 業務規則

1. 配送服務只接受已啟用店家與取貨據點。
2. 建立訂單前須檢查航班時間是否滿足配送所需安全時間。
3. 未付款訂單不得進入正式交寄流程；是否允許店家代墊待確認。
4. 實際付款金額必須與平台訂單金額一致。
5. 同一筆金流回呼及物流事件必須可重複接收，但只能處理一次。
6. 已交寄訂單不可直接刪除，只能依流程取消、退款或退運。
7. 取貨需同時驗證取貨憑證及旅客身分；代理領取規則待確認。
8. 超過保管期限未取貨時轉為逾期未取，後續依政策退運或處置。
9. 所有費用、狀態、退款與人工更正均需保留歷程，不得直接覆蓋而無紀錄。

## 11. 例外與風險情境

| 情境 | 建議處理方式 |
| --- | --- |
| 付款失敗或逾時 | 保留待付款狀態，允許重新產生付款交易 |
| 金流重複回呼 | 以交易編號與事件識別碼進行冪等處理 |
| 付款成功但平台未更新 | 由排程主動查詢交易並補償更新 |
| 物流 API 暫時失敗 | 重試並記錄失敗；提供人工補登功能 |
| 航班提前或異動 | 由客服調整取貨資料並留下異動紀錄 |
| 無法在起飛前送達 | 通知旅客與客服，依政策取消、退款或改送地點 |
| 包裹破損、遺失或件數不符 | 建立異常案件，保留照片、責任與處理結果 |
| 旅客未取貨 | 到期提醒，之後轉入退運或保管費流程 |
| 取貨碼外洩 | 仍須核對護照；必要時作廢並重發取貨碼 |

## 12. 非功能需求

### 12.1 安全與個資

- 全程使用 HTTPS。
- 角色與資料範圍權限須由伺服器端驗證。
- 密碼使用安全雜湊，不得明碼保存。
- API 金鑰、金流密鑰不得寫入原始碼。
- 護照與聯絡資料應加密、遮罩並設定保存期限。
- 查詢與匯出敏感資料需留下稽核紀錄。
- 正式上線前需確認台灣個資法、金流及跨境旅客資料相關要求。

### 12.2 可用性與效能

- MVP 建議一般查詢在正常負載下 3 秒內回應。
- 金流與物流介接失敗不得造成訂單資料遺失。
- 關鍵背景作業需具備重試、告警及人工補償機制。
- 定期備份資料庫，並演練還原流程。

### 12.3 多語系

- 旅客介面至少支援繁體中文與英文。
- 介面文字、通知範本及錯誤訊息不可硬寫於程式碼，需可維護與翻譯。
- 日期、時間、幣別與姓名格式需考量外籍旅客使用情境。

## 13. 建議系統架構

MVP 建議採模組化單體架構，降低初期部署及維運複雜度，同時保留後續拆分能力。

- 前端：店家／後台 Web；旅客可先採響應式 Web，不必立即開發 App。
- 後端：ASP.NET Core Web API（C#）。
- 資料庫：SQL Server。
- 背景工作：處理金流補查、物流同步、通知及失敗重試。
- 外部介接層：金流與中華郵政各自封裝 Adapter，避免外部格式滲入核心業務。
- 記錄與監控：集中式應用程式日誌、介接追蹤碼與異常告警。

建議依領域區分 `Orders`、`Payments`、`Shipments`、`Pickups`、`Notifications`、`Identity` 模組；共用資料應透過明確介面存取，避免所有功能直接互相修改資料表。

## 14. MVP 驗收標準

1. 已授權店家人員可建立一筆資料完整的配送單。
2. 系統能依確認後規則產生正確應付金額。
3. 測試金流付款成功後，訂單只更新一次且留有交易紀錄。
4. 付款失敗可重新付款，不會產生重複有效付款。
5. 店家可登錄唯一託運單號並將訂單轉為配送中。
6. 機場人員可完成到貨點收及存放位置登錄。
7. 旅客可取得取貨憑證並查詢訂單狀態。
8. 機場人員核對身分與一次性取貨碼後可完成取貨；同一取貨碼不可再次使用。
9. 管理者可查詢完整訂單、付款、物流與狀態歷程。
10. 未授權使用者無法查閱或操作其他店家／據點資料。

## 15. 待確認事項

### 15.1 高優先，會直接影響 MVP

1. 配送單由店家、旅客或兩者共同建立？
2. 首期服務哪些店家、機場、航廈及取貨櫃檯？
3. 「通關金流」的正式服務商名稱與 API 規格為何？
4. 中華郵政是否已有企業合約及可使用的 API？
5. 運費、服務費、稅額及幣別的計算規則為何？
6. 航班起飛前至少需預留多少配送與取貨時間？
7. 哪些商品禁止配送？由誰執行檢查並承擔責任？
8. 取貨時必須核驗哪些資料？是否允許代理領取？
9. 退款、退運、遺失、破損及逾期未取的責任與費用如何處理？
10. 旅客介面與通知需支援哪些語言？

### 15.2 次要，可於第二階段確認

1. 是否需要旅客會員帳號或僅用訂單查詢？
2. 是否需要簡訊、LINE、WhatsApp 或其他通知管道？
3. 是否需要串接航班即時資訊？
4. 是否需要店家、物流商及平台間的分潤與自動對帳？
5. 是否需要手機 App？

## 16. 建議開發順序

1. 確認高優先待確認事項及狀態流程。
2. 完成配送單、旅客、航班、包裹及費用資料模型。
3. 完成店家建單、後台查詢及權限控制。
4. 串接金流並完成回呼、補查與重複事件防護。
5. 完成郵政託運單號與物流狀態功能。
6. 完成機場到貨、取貨憑證與核銷。
7. 完成通知、異常處理及基本報表。
8. 執行端到端驗收、資安檢查與上線演練。

- 金流與物流狀態另建對照表，避免散落於程式碼。
- API 規格建議使用 OpenAPI；資料庫結構、流程圖與畫面原型納入同一版本庫。
- 需求確認後，再產出畫面規格、API 規格、ERD、狀態轉換表及測試案例。

## 18. 結論

此平台的 MVP 核心不是單純「寄送查詢」，而是確保一筆配送單能安全地完成「建單、付款、交寄、運送、機場點收、身分核驗、取貨」的完整閉環。現階段最重要工作是取得金流及中華郵政介接資料，並確認航班時效、禁運品、取貨與異常處理規則；上述內容確認後，才適合進入正式畫面、API 與資料庫設計。
