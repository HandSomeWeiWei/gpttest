# 配送平台待確認事項

以下問題會影響契約或資料一致性，確認前不應由程式自行推定。

| ID | 優先級 | 問題 | 建議決策 |
|---|---|---|---|
| Q-001 | 阻擋 | 正式／測試 Base URL 與 `WebName` 尚未填寫 | 由部署環境設定提供，不寫死於程式 |
| Q-002 | 阻擋 | 多個 JSON 範例缺少逗號，且 `value_amt` 無值 | 修正成可通過 JSON Schema 的範例 |
| Q-003 | 阻擋 | `sendDeliveryDetails` 前置條件將 `order_status` 與 `payment_status` 都比對 `paid`，但後續又把訂單設為 `finish` | 定義三套狀態的正式代碼與狀態圖 |
| Q-004 | 高 | `cancelDelivery` 成功後訂單轉 `return` 或 `voided`，語意與退款／退運容易混淆 | 明確定義 cancel、return、refund、voided |
| Q-005 | 高 | API 回應有 `ret_code`，部分範例／邏輯又使用 `auth_result`、`proc_status` | 建立每層代碼對照表及 HTTP status 規則 |
| Q-006 | 高 | 文件同時出現 `payment_status_anme`、`location_nam` 等拼字 | 對外契約統一為 `payment_status_name`、`location_name`，確認相容策略 |
| Q-007 | 高 | SQL 多處重複指定 `modify_date`，第二個值實際應為修改者欄位 | 確認欄名，例如 `modify_user` |
| Q-008 | 高 | `queryOrderDetail` JOIN 條件 `ob.box_no = ob.box_no` 永遠成立 | 應確認是否為 `ob.box_no = sb.box_no` 並加店家條件 |
| Q-009 | 高 | 郵件號碼產生規則仍標記待中華郵政提供 | 確認由平台預先產生或物流 API 回傳 |
| Q-010 | 高 | 退款鎖定時間在流程圖為晚上 8 點至隔天 10 點，API 文件為隔天 8 點 | 由業務確認單一時段，改為可設定參數 |
| Q-011 | 中 | 日期欄位混用 ISO 8601、`yyyyMMdd`、`yyyyMMddHHmmss`，未說明時區 | API 統一 ISO 8601 +08:00；外部格式在介接層轉換 |
| Q-012 | 中 | `user_no` 同時放在 JWT `sub` 與 request body | 以 JWT 為權威並檢查兩者一致，或移除 body 欄位 |
| Q-013 | 中 | 退款、預約、取消缺乏冪等鍵與重送規則 | 以訂單號＋操作類型建立唯一操作紀錄 |
| Q-014 | 中 | 流程圖的發票開立、中獎通知、代收證明 email 尚無完整 API／排程規格 | MVP 標示不實作，另開後續規格 |
| Q-015 | 中 | 個資保存期限、遮罩、稽核與刪除規則未定義 | 由資安／法遵補充資料生命週期 |

## 更新方式

確認後請保留 ID，在表格新增「決議」與「決議日期」，並同步修改 `配送平台設計規格.md`、API Schema 及測試案例。
