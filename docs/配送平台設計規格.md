# 配送平台設計規格（Codex 開發參考）

> 狀態：MVP 草案  
> 來源：`source/宅配API規格書_20260709.docx`、`source/系統串接流程-先列印託運單版本.drawio`  
> 原則：來源未定義的行為不得自行推定，請先查閱[待確認事項](待確認事項.md)。

## 1. 目標與範圍

配送平台承接 e 指購建立的退稅商品與帳單資料，讓店家選擇物流業者、箱型與配送地點，建立訂單及配送單；旅客可透過行動裝置付款，平台再串接通關金流與中華郵政，處理預約配送、狀態查詢、取消、退款與退運。

MVP 包含：身分驗證、物流基礎資料、訂單建立與查詢、配送明細、付款結果查詢、配送預約與取消、配送狀態、訂單退款與取消。發票開立、中獎通知與代收證明郵件屬流程圖需求，但細部 API 尚未完整定義。

## 2. 系統角色與邊界

| 角色／系統 | 責任 |
|---|---|
| e 指購（商店端） | 分箱挑選退稅商品、選擇物流與配送資料、建立及查詢訂單、列印託運單 |
| 配送平台 API | 驗證、訂單與配送資料管理、金流／物流介接 |
| 配送平台 Web | 旅客註冊、條款同意、付款與結果顯示 |
| 配送平台排程 | 配送狀態同步及通知；頻率與重試策略待定 |
| 通關金流 API | 帳單上傳、付款授權、授權查詢與退款 |
| 中華郵政 API | 出貨明細、取消出貨與郵件狀態查詢 |

## 3. 主流程

1. 店員登入 e 指購並完成退稅商品分箱。
2. e 指購呼叫 `loginAuth` 取得 JWT。
3. 查詢物流業者及配送相關資料，選擇箱型、配送地點與內容物。
4. 呼叫 `insertOrder` 建立發票／配送訂單。
5. 呼叫 `insertDeliveryItems` 建立配送明細及郵件號碼，列印託運單。
6. 旅客掃描付款 QR Code，在配送平台 Web 完成付款。
7. e 指購以 `queryAuth` 查詢授權結果；成功後可呼叫 `sendDeliveryDetails` 預約配送。
8. 平台或排程以 `queryDeliveryStatus` 查詢最新物流狀態。
9. 未交由物流業者收件前，可依狀態執行配送取消、訂單取消或退款。

## 4. API 共通契約

- 傳輸：HTTPS、JSON、UTF-8，`Content-Type: application/json`。
- HTTP 動詞：全部使用 `POST`。
- 驗證：除 `loginAuth` 外，Header 必須包含 `Authorization: Bearer {token}`。
- JWT：`sub` 為使用者編號，`exp` 為 Unix timestamp；原規格設定登入後 4 小時到期。
- 回應外層：`ret_code`、`message`、`result`。
- 共通成功碼：`0000`；資料檢核 `W001`；Token 無效 `W002`；Token 過期 `W003`；系統錯誤 `E999`。
- 金額欄位應使用 decimal，不可使用浮點數。時間、時區與日期格式須在實作前確認。
- API 必須具備結構化紀錄與 correlation ID；不得記錄密碼、完整 JWT 或敏感個資。

## 5. API 功能清單

| # | API | 用途 | 主要前置條件 |
|---:|---|---|---|
| 1 | `loginAuth` | 驗證 e 指購／Web 帳號並取得 JWT | e 指購驗證網址與店家資料有效 |
| 2 | `getLogisticProvider` | 取得啟用中的物流業者 | JWT 有效 |
| 3 | `getDeliveryInfo` | 取得店家、箱型、費率、配送地點、內容物與手續費 | 店家及物流業者有效 |
| 4 | `queryAuth` | 查詢一或多筆付款授權結果 | 訂單存在 |
| 5 | `insertOrder` | 建立訂單、箱子及商品明細 | 金額與箱項資料通過檢核 |
| 6 | `insertDeliveryItems` | 建立寄收件與材積明細、產生郵件號碼 | 訂單與箱項存在 |
| 7 | `queryOrder` | 依日期及護照號碼查詢訂單 | 查詢條件合法 |
| 8 | `queryOrderDetail` | 查詢訂單、商品及配送明細 | 訂單存在 |
| 9 | `queryDeliveryStatus` | 查詢每一郵件號碼最新狀態 | 配送明細存在 |
| 10 | `sendDeliveryDetails` | 傳送出貨明細並預約配送 | 已付款、尚未預約 |
| 11 | `cancelDelivery` | 通知物流業者取消配送 | 已預約且仍可取消 |
| 12 | `refundOrder` | 向通關金流申請退款 | 已付款且不在結帳鎖定期間 |
| 13 | `cancelOrder` | 取消未付款、未預約的訂單 | 訂單狀態為 `create` |

欄位、範例、SQL 與逐項處理邏輯請參考 [API 規格原文轉錄](API規格原文轉錄.md)。實作前必須先處理[待確認事項](待確認事項.md)中的契約錯誤。

## 6. 核心資料

| 資料群組 | 主要資料表 | 用途 |
|---|---|---|
| 使用者與店家 | `FD_user`、`FD_TaxRefundStore` | 帳號、店家與 JWT 金鑰 |
| 物流主檔 | `FD_Logistics`、`FD_LogisticBox` | 物流業者、特約戶與費率 |
| 店家箱型 | `FD_StoreBox` | 店家可選箱型及尺寸 |
| 配送地點 | `FD_DeliveryLocation` | 可選收件／取件地點 |
| 訂單 | `FD_Order` | 訂單、付款、配送狀態 |
| 訂單箱與商品 | `FD_OrderBox`、`FD_OrderBoxProduct` | 分箱與商品明細 |
| 配送明細 | `FD_DeliveryItem` | 寄收件人、材積、郵件號碼與上傳批號 |
| 參數 | `FD_Parameter` | 狀態名稱、內容物、時段、手續費與驗證金鑰 |

所有跨表新增／更新必須使用資料庫交易。外部 API 不應包在長交易中；採「本地待處理 → 呼叫外部服務 → 依結果更新」並提供冪等鍵與重試紀錄。

## 7. 狀態與操作規則

原規格同時使用 `order_status`、`payment_status`、`shipping_status`，實作時必須分開保存，不可合併為單一狀態。

| 操作 | 訂單條件 | 付款條件 | 配送條件 | 成功後摘要 |
|---|---|---|---|---|
| 預約配送 | 規格處出現 `paid`／`finish` 的不一致，待確認 | `paid` | `pending` | 訂單 `finish`、配送 `shipped` |
| 配送取消 | `finish` 或 `refund` | 未明定 | `shipped` | 訂單轉 `return` 或 `voided`、配送 `cancel` |
| 退款 | `paid`、`finish` 或 `return` | `paid` | 視訂單狀態 | 付款 `refund`；訂單轉 `refund` 或 `voided` |
| 訂單取消 | `create` | 應為未付款 | 應為未預約 | 訂單 `cancel` |

狀態轉移需集中在 Domain Service／State Machine，禁止各 Controller 自行更新狀態。

## 8. 非功能需求

- 安全：HTTPS、密碼採 BCrypt、密鑰不得寫入程式碼；個資欄位須遮罩與權限控管。
- 冪等：建立訂單、建立配送明細、預約、取消、退款必須防止重送造成重複交易。
- 可觀測性：記錄 API 名稱、訂單號、結果碼、耗時、外部服務結果與 correlation ID。
- 韌性：外部服務逾時、重試次數、退避及人工補償機制應可設定。
- 一致性：外部呼叫成功但本地更新失敗時，必須產生待補償工作，不可只回傳一般錯誤。
- 測試：每支 API 至少包含成功、必填、長度、Token、狀態不符、外部服務失敗與重送案例。

## 9. MVP 驗收條件

1. 店家能完成登入並取得可驗證、可過期的 JWT。
2. 能取得有效物流、箱型、費率、地點與內容物資料。
3. 同一請求重送不會重複建立訂單、郵件號碼、預約或退款。
4. 已付款訂單可預約配送；不符狀態時回傳明確業務錯誤。
5. 配送可在物流允許時取消，失敗原因可追蹤到個別箱項／郵件號碼。
6. 可查詢訂單明細與每件郵件的最新狀態。
7. 退款與取消只能在允許的狀態執行，並保留稽核紀錄。
8. 自動化測試覆蓋核心狀態轉移及外部服務失敗補償。

## 10. Codex 實作規則

1. 修改前先讀本文件、[API 規格原文轉錄](API規格原文轉錄.md)與[待確認事項](待確認事項.md)。
2. 原文件與本文件衝突時，不得猜測；新增待確認項目並停止受影響功能的實作。
3. Controller 僅負責 HTTP 契約；驗證、狀態轉移與外部介接分層處理。
4. API DTO、資料庫 Entity、外部服務 DTO 必須分離，避免外部契約變更污染資料模型。
5. 每次狀態或 API 契約變更，必須同步更新本文件及測試。
